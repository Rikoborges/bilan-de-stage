C'est quoi WooCommerce ?

WooCommerce est un plugin WordPress qui transforme un site normal en boutique en ligne.

WordPress tout seul gère les contenus — articles, pages, médias. WooCommerce ajoute par-dessus une couche e-commerce complète : produits, panier, commandes, paiements, livraisons. Techniquement, WooCommerce utilise les mécanismes natifs de WordPress — les Custom Post Types — pour stocker les produits comme des "posts" spéciaux dans la base de données.

La différence clé avec un site classique : un client peut ajouter au panier, passer une commande et payer en ligne. WooCommerce gère toute la logique de transaction.

WordPress seul

Pages, articles, médias. Contenu statique ou dynamique. Pas de transaction commerciale.

→ site vitrine

WordPress + WooCommerce

Tout WordPress + produits, panier, commandes, paiements, stock, livraisons.

→ boutique en ligne

Pages créées automatiquement : Boutique, Panier, Commander, Mon compte. WooCommerce les crée lors de l'installation. Ne jamais les supprimer — WooCommerce en a besoin pour fonctionner.

Produits simples et variables

Les deux types de produits les plus courants dans une boutique.

Un produit simple est le cas de base — un seul article avec un prix fixe. Un produit variable a des variations : tailles, couleurs, capacités... Chaque variation peut avoir un prix différent. On définit d'abord des attributs (ex: Taille), puis des variations (S, M, L) avec chacune son prix et son stock.

Le SKU (Stock Keeping Unit) est une référence unique pour identifier un produit dans le stock — utile pour la gestion d'inventaire et l'intégration avec des systèmes externes (ERP, logistique...).

Produit simple

Un seul prix, un seul stock. Ex : un livre, une bougie, un service à prix fixe.

type: simple

Produit variable

Attributs + variations avec prix différents. Ex : T-shirt S/M/L, chaque taille a son prix.

type: variable
Exemple — produit variable T-shirt
Attribut :   Taille
Valeurs :    S | M | L

Variations :
  Taille S  →  prix : 15,00€  |  stock : 10  |  SKU : TS-S-001
  Taille M  →  prix : 15,00€  |  stock : 8   |  SKU : TS-M-001
  Taille L  →  prix : 17,00€  |  stock : 3   |  SKU : TS-L-001

// Le client choisit sa taille → prix et stock mis à jour

Description courte vs longue : la courte s'affiche à côté du prix (résumé rapide). La longue s'affiche dans un onglet en bas de la page produit (détails complets).

Structure technique WooCommerce

Comment WordPress stocke les produits en base de données.

Dans WordPress, tout est un "post". Les articles sont des posts de type post. Les pages sont de type page. WooCommerce crée un Custom Post Type appelé product. Un produit est stocké dans la table wp_posts avec le type product, et ses données spécifiques (prix, stock, SKU) sont stockées dans wp_postmeta sous forme de métadonnées.

Les catégories de produits sont des taxonomies personnalisées (product_cat), comme les catégories d'articles WordPress — le même système, juste avec un nom différent.

base de données — tables impliquées
── wp_posts ──────────────────────────────────────────────
  ID: 42  |  post_type: 'product'  |  post_title: 'T-shirt blanc'
  post_status: 'publish'  |  post_author: 1

── wp_postmeta (métadonnées du produit) ──────────────────
  post_id: 42  |  meta_key: '_price'      |  meta_value: '15.00'
  post_id: 42  |  meta_key: '_stock'      |  meta_value: '10'
  post_id: 42  |  meta_key: '_sku'        |  meta_value: 'TS-001'
  post_id: 42  |  meta_key: '_sale_price' |  meta_value: '12.00'

// wc_get_product(42) assemble tout ça en un objet WC_Product

Pourquoi c'est utile de savoir ça : quand on cherche un bug ou qu'on veut faire une requête personnalisée, savoir où les données sont stockées évite des heures de recherche.

Hooks — add_action et add_filter

Personnaliser WooCommerce depuis un plugin, sans jamais modifier les fichiers du thème.

WooCommerce (comme WordPress) fournit des centaines de points d'accroche (hooks) dans son code. À chaque moment du chargement d'une page, WooCommerce appelle ces hooks. On peut "accrocher" nos propres fonctions pour modifier le comportement sans toucher au code source. Si on modifie directement les fichiers du thème, une mise à jour efface tout. Les hooks dans un plugin survivent à toutes les mises à jour.

add_action = on ajoute du contenu à un moment précis du rendu. add_filter = on intercepte une valeur et on la modifie avant qu'elle soit affichée.

hooks-woocommerce.php
// ── add_action : ajouter du contenu ─────────────────────
// Ajouter un message juste sous le titre de la page produit
add_action(
    'woocommerce_single_product_summary',  // où (nom du hook WC)
    function() {
        echo '<p class="livraison">🚚 Livraison gratuite dès 50€</p>';
    },
    6  // priorité : 5 = titre, 10 = prix — on s'insère à 6
);

// ── add_filter : modifier une valeur ─────────────────────
// Ajouter "TTC" après chaque prix affiché par WooCommerce
add_filter(
    'woocommerce_get_price_html',   // quelle valeur intercepter
    function( $price, $product ) {
        return $price . ' <small>TTC</small>';
    },
    10, 2  // priorité 10, reçoit 2 arguments
);

Comment trouver les hooks WC : chercher do_action( ou apply_filters( dans les fichiers source de WooCommerce, ou consulter la référence officielle sur woocommerce.github.io/code-reference.

Lire WC_Product et créer un shortcode

Accéder aux données d'un produit WooCommerce depuis un plugin PHP.

WC_Product est une classe PHP — un objet qui représente un produit avec toutes ses données. wc_get_product() crée cet objet à partir de l'ID du produit. On appelle ensuite ses méthodes pour lire les données : get_name(), get_price(), get_permalink()...

wc_get_products() retourne un tableau de plusieurs objets WC_Product selon des critères (limite, date, catégorie...). On boucle avec foreach pour traiter chaque produit. C'est la solution qu'on a utilisée au stage pour contourner le problème CORS — plus besoin d'appel HTTP externe.

shortcode-produits-recents.php
function produits_recents() {

    // Récupérer les 3 derniers produits publiés
    // Pas de requête HTTP — on appelle directement WooCommerce
    $produits = wc_get_products([
        'limit'   => 3,
        'orderby' => 'date',
        'order'   => 'DESC',
        'status'  => 'publish',
    ]);

    $html = '<ul class="produits-recents">';

    // Boucler sur chaque objet WC_Product
    foreach ( $produits as $p ) {
        $html .= '<li>'
               . '<a href="' . $p->get_permalink() . '">'
               . $p->get_name()        // nom du produit
               . ' — '
               . $p->get_price_html()  // prix formaté avec €
               . '</a></li>';
    }

    return $html . '</ul>';
}

// Utilisation dans une page WP : [produits_recents]
add_shortcode( 'produits_recents', 'produits_recents' );

API REST WooCommerce

WooCommerce expose ses données via une API REST — comme n'importe quelle API externe.

En plus des fonctions PHP, WooCommerce fournit une API REST complète. On peut accéder aux produits, commandes et clients via des URLs HTTP — exactement comme on l'a fait avec l'API Chuck Norris en Partie 1. C'est utile pour connecter la boutique à une application mobile, un autre site, ou un outil tiers.

C'est là que le problème CORS s'est posé au stage — en essayant d'appeler /wp-json/wc/v3/products avec fetch() depuis LocalWP. La solution qu'on a trouvée : utiliser wc_get_products() directement en PHP — on contourne entièrement le problème.

Endpoints disponibles
// Liste de tous les produits
GET https://monsite.com/wp-json/wc/v3/products

// Un produit spécifique par son ID
GET https://monsite.com/wp-json/wc/v3/products/42

// Les commandes (authentification obligatoire)
GET https://monsite.com/wp-json/wc/v3/orders

// Réponse JSON exemple — même format qu'une API externe :
[
  { "id": 42, "name": "T-shirt blanc", "price": "15",
    "permalink": "https://monsite.com/produit/t-shirt-blanc",
    "stock_quantity": 10, "sku": "TS-001" }
]

Pour les données sensibles : générer des Consumer Key / Consumer Secret dans WooCommerce → Réglages → Avancé → API REST et les envoyer dans les headers de la requête. Ne jamais les mettre dans le code front-end.
